## 1. What Weeber is, in one paragraph

Self-hosted personal cloud storage. The user's own computer (Mac / Win / Linux)
is the "host" — it holds the files on disk. Phones, tablets, other computers
are "clients" that reach those files through a VPS-backed relay (persistent
outbound WebSocket from the host so no router config is ever needed). Single
account can have many clients but exactly one active host (WhatsApp-model).
Lifetime one-time pricing; cloud backup is the only recurring SKU.

## 4. Architecture — what calls what

```
┌──────────────┐  WSS  ┌────────────┐  HTTPS  ┌──────────────┐
│  HOST (Mac)  │──────▶│   VPS      │◀────────│ CLIENT (phone│
│              │       │            │         │ / browser /  │
│  Files on    │       │ weeber-api │         │ demoted Mac) │
│  local disk  │       │ weeber-web │         │              │
│  FileIndex   │       │ SQLite DB  │         │ RelayClient  │
│  FileStorage │       │ TunnelHub  │         │              │
│  HostTunnel  │       │            │         │              │
└──────────────┘       └────────────┘         └──────────────┘
        ▲                    ▲
        │                    │
        │                    ▼ DNS (future)
        └──── outbound only ─────── weeber.app (future)
```

Bytes path for phone upload:
`phone → POST /api/files → Next.js → POST /v1/relay/upload → tunnel WS frame
 → host → FileStorage.write + FileIndex.insert → res frame back → 200 to phone`

Download is the reverse; host chunks response body into ≤4 MB WS frames so
large files don't hit the `@fastify/websocket maxPayload: 64 MB` ceiling.

## 5. Auth model (industry-standard rotating refresh tokens)

| Token | TTL | Storage | Purpose |
|---|---|---|---|
| **Access JWT** | 1 hour | In-memory + httpOnly cookie / Flutter secure storage | Bearer on every API call |
| **Refresh token** | 90 days, **rotating one-shot** | Server-hashed row (SHA-256); client gets the raw in httpOnly cookie / secure storage | Swap for a new pair |

- `POST /v1/auth/login` → both tokens
- `POST /v1/auth/refresh { refresh_token }` → rotates + issues new pair
- `POST /v1/auth/logout { refresh_token }` → revokes token + family
- `POST /v1/devices` → device-bound pair (device_id in JWT `did` claim)
- `DELETE /v1/devices/:id` → also revokes every refresh token for that device

**Theft detection** (OAuth 2.1 spec): re-using an already-used refresh token
revokes the whole family. Tested in prod, returns `{"error":"reuse"}`.

**Client auto-refresh:**
- Website: `lib/session.ts` proactively rotates when JWT exp < 60 s.
  `lib/authFetch.ts` reactively retries any 401.
- Flutter: `AuthNotifier.refreshIfNeeded()` is the single entrypoint.
  `RelayClient._withAuth()` wraps every request with 401 → refresh → retry.
  `HostTunnel` catches WS `{type:'error', error:'invalid_token'}` frames and
  refreshes before reconnecting.

## 8. Capacity planning

### Current 1.9 GB / 1-core VPS
- With today's client polling: safe to **~3,000–4,000 total users** at 8%
  concurrency, hard cap ~500 concurrent active.
- Network bandwidth usually binds first (100 Mbps shared).

### Path to 8 GB / 4-core (user has this on standby)
- Polling model: ~2,500–3,500 concurrent, **~30K–45K total users**
- After push migration: ~10–20K concurrent, **~125K–250K total**
- Migration is 20 minutes of rsync + DNS flip if a domain is in place.

### Push-vs-polling migration (when crossing ~500 concurrent)
Detailed plan in `memory/weeber_polling_at_scale.md`. Rough shape:
1. Host pushes file-change deltas to its tunnel WS
2. VPS forwards to subscribed browsers via SSE/WS
3. Client side: one persistent connection receives events; zero idle cost
4. Polling removed or gated to "visibility fallback"

Roughly a 1-day build.
